# ROADMAP 종결 및 차기 분리 보고서

## 📝 작업 요약
- 현재 live repo를 기준으로 `ROADMAP.md`를 재판정한 결과, 현 세대(2026-02~2026-03) 항목은 이미 전부 닫혀 있었고 남은 문제는 구현 누락이 아니라 **문서 종결 방식**이었다.
- 이번 세션에서는 `ROADMAP.md`를 active backlog가 아닌 **닫힌 snapshot 문서**로 명시하고, genuinely future-facing 항목만 `NEXT_ROADMAP.md`로 분리했다.
- 추가로 남아 있던 문서 충돌 2건을 정리했다: `README.ko.md` CRC/Counter 제한사항 문구, `tests/oracle/ORACLE_RESULTS.md`의 historical snapshot framing.

## 🛠 변경 상세

### 1) `ROADMAP.md` 종결 처리
- 상단에 close-out 배너를 추가해 이 문서가 더 이상 active backlog가 아님을 명시했다.
- 현 세대 범위를 `2026-02 ~ 2026-03` snapshot으로 고정했다.
- 실제 존재 경로에 맞게 근거 문서 경로를 `analysis/Codebase_Analysis.md`로 정정했다.
- 하단의 읽기 순서를 현재 canonical source hierarchy에 맞게 재정리했다.
- `Oracle 후속 backlog` 섹션은 active backlog가 아니라 **본 세대에서 종결된 후속 항목 기록**임을 명확히 했다.
- `권장 착수 순서`는 현재 지시가 아니라 historical note임을 드러내도록 라벨을 바꿨다.

### 2) 차기 작업 분리
- 새 파일 `NEXT_ROADMAP.md`를 생성했다.
- 현재 ROADMAP에 다시 넣으면 closed snapshot을 흔들게 되는 future-facing 항목만 이관했다.
- 이번 세션에서 분리한 항목:
  - `>64` 신호 valid 비트마스크 fallback 정책(배열 기반 도입 여부)
  - oracle reference decoder 비호환 DBC 대응 전략

### 3) 문서 정합성 보정
- `README.ko.md`
  - 영어 README와 맞지 않던 CRC/Counter 제한사항 문구를 현행 shipped scope로 정정했다.
  - 자동 heuristic validation 미구현과 explicit metadata 기반 generated handling 지원을 분리해 명시했다.
- `tests/oracle/ORACLE_RESULTS.md`
  - 현재 canonical status가 아니라 2026-03-12 시점 snapshot임을 상단 경고로 명시했다.

### 4) source-of-truth hierarchy
- 현 세대 roadmap close-out 이후 우선순위는 아래와 같이 읽는 것이 가장 안전하다.
  1. 최신 `Reports/` 검증/종결 문서
  2. `ROADMAP.md` (닫힌 historical snapshot)
  3. `NEXT_ROADMAP.md` (active future backlog)
  4. `tests/oracle/CATEGORY_C_EXCEPTIONS.md`
  5. `tests/oracle/ORACLE_RESULTS.md` (dated snapshot)
